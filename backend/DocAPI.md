## 1. Overview

Vault Backend is a REST API built with **FastAPI** and **PostgreSQL** (async via SQLAlchemy + asyncpg). It implements a password management system shared among user groups using **End-to-End Encryption (E2E)**.

The server **never sees** the group passwords in plaintext nor the user's master password. All shared secret cryptography happens on the client side. The server only stores encrypted blobs and hashes.

### Key Concepts

| Concept | Description |
| --- | --- |
| **user_id** | The user's phone number. Used as an external identifier. |
| **id_lookup** | `HMAC-SHA256(user_id, PEPPER)` — a deterministic value indexed for O(1) searches. |
| **id_hash** | `Argon2id(user_id + PEPPER)` — defense-in-depth verification after the lookup. |
| **password_hash** | `Argon2id(SHA256(password) + PEPPER2)` — slow password hash. The client sends the SHA-256; the server never sees the actual password. |
| **SGK** | Shared Group Key — symmetric key (e.g., AES-256) generated by the client for each group. It is encrypted with each member's public key before being uploaded. |
| **public_key** | User's EC public key (P-256). Used to encrypt the SGK for that user via ECIES. |

## 2. Security Model

### User Identification

1. The client sends the **phone number** as `user_id`.
2. The server calculates `HMAC-SHA256(user_id, PEPPER)` → `id_lookup` column (indexed, O(1) search).
3. As defense-in-depth, the server also stores `Argon2id(user_id + PEPPER)` → `id_hash` column. After finding the user via HMAC, it verifies with Argon2 to rule out collisions.

### Master Password (Zero-Knowledge)

1. The client calculates `SHA-256(password)` and sends the 64-character hex digest as `password_hash`.
2. The server applies `Argon2id(password_hash + PEPPER2)` and stores the result.
3. The server **never sees the real password** — it only receives the SHA-256 hash.
4. `PEPPER2` is an internal server secret, **different** from `PEPPER`.

### Shared Secrets (E2E)

* The server **has no access** to Group Keys (SGK) or shared passwords. It only stores encrypted blobs.
* Each group member receives the SGK encrypted with their own public key.
* Group passwords are encrypted with the SGK (e.g., AES-256-GCM) on the client side.

### Session Tokens

* **JWT** signed with HS256.
* The `sub` (subject) of the token is the internal UUID of the user (never exposed as input data).
* Configurable expiration (default: 60 minutes).

## 3. Authentication (JWT)

All endpoints (except `/health`, `/auth/register`, and `/auth/login`) require a JWT token in the header:

```
Authorization: Bearer <access_token>

```

---

## 4. Endpoints

### 4.1. Health check

`GET /health` | No Auth. Returns `{"status": "ok"}`.

---

### 4.2. Auth

#### 4.2.1. Register

`POST /auth/register`

| Field | Type | Description |
| --- | --- | --- |
| `user_id` | `string` | Phone number. |
| `password_hash` | `string` | SHA-256 hex-digest of the master password. |
| `public_key` | `string` | User's public key (PEM/base64). |

#### 4.2.2. Login

`POST /auth/login`
Returns the `access_token`.

#### 4.2.3. Public key lookup

`GET /auth/public-key/{user_id}`
Used to retrieve another user's public key to encrypt an SGK for them.

---

### 4.3. Groups

#### 4.3.1. Create group

`POST /groups/`

* `name`: Group name.
* `encrypted_sgk`: SGK encrypted with the creator's public key.

#### 4.3.2. List my groups

`GET /groups/`
Lists groups and includes the user's specific `encrypted_sgk`.

#### 4.3.3. Add member

`POST /groups/{group_id}/members`
Requires `user_id` (phone) and the `encrypted_sgk` (encrypted with the new member's public key).

---

### 4.4. Passwords

#### 4.4.1. Create password

`POST /groups/{group_id}/passwords`

* `label`: Name/tag for the entry.
* `encrypted_data`: Password data encrypted with the Group SGK (base64).

#### 4.4.2. List passwords

`GET /groups/{group_id}/passwords`
Returns all encrypted entries. The client must decrypt them locally.

---

## 6. Typical Usage Flow

1. **Registration:** User sends phone, SHA-256 of password, and public key.
2. **Login:** User authenticates and receives a JWT.
3. **Create Group:** User generates an SGK (AES key), encrypts it with their own public key, and creates the group.
4. **Add Member:** Fetch the new member's public key, encrypt the SGK with it, and upload.
5. **Share Password:** Encrypt the password with the SGK and upload to the group.
6. **Read Password:** Member retrieves their copy of the SGK, decrypts it with their private key, and then decrypts the group passwords.

## 7. Server Configuration

| Variable | Description | Default |
| --- | --- | --- |
| `DATABASE_URL` | PostgreSQL connection URL | `postgresql+asyncpg://...` |
| `SECRET_KEY` | JWT signing key | `change-me` |
| `PEPPER` | Secret for ID HMAC/Argon2 | `change-me` |
| `PEPPER2` | Secret for Password Argon2 | `change-me-too` |
